# Bump 验证协议 / Bump Validation Protocol

> 原则 #2 的完整规范。rubric 升级时，新旧公式必须在历史样本上通过一致性检验。

## 核心规则

1. rubric 升级时，校准池所有有实绩数据的样本必须用新公式重打分
2. 新排序与实际表现排序若在 ≥4/5 样本上不一致，升级被拒
3. 升级必须经跨模型独立审核（默认要求，`CROSS_MODEL_AUDIT=false` 时自审）
4. 改 THRESHOLD 本身算元层级 bump，单独走流程

## 公众号长文的 Bucket 体系

基于公众号后台可获取的数据，定义阅读量 bucket：

| Bucket | 阅读量范围 | 含义 |
|---|---|---|
| S | >50,000 | 爆款 |
| A | 10,000-50,000 | 热文 |
| B | 3,000-10,000 | 中上 |
| C | 1,000-3,000 | 基准线 |
| D | 300-1,000 | 低于均值 |
| E | <300 | 冷启动 |

> 注意：bucket 边界会随账号体量变化。有初始粉丝的用户（如 4000 关注），基准线应上移。grindraft-init 会根据用户提供的粉丝数调整初始 bucket。

## 升级流程

1. **触发条件**（grindraft-retro Phase 7 检测）：
   - 连续 ≥3 次同向偏差（高估/低估）
   - 或 1 次极端偏差（≥5x 偏离中枢）
   - 或跨维度 pattern 出现

2. **全量重打**：
   - 用新公式对校准池所有样本重新计算 composite
   - 新 composite 排序 vs 实际阅读量排序
   - Rank correlation（Spearman/Kendall）在 ≥4/5 样本上一致 → 通过

3. **跨模型审核**：
   - 将新公式 + 5 个样本的稿子内容发给另一个模型（独立 session）
   - 对方独立打分，对比一致性
   - 差异 ≥2 的维度超过 2 个 → 升级被拒

4. **Cleanup pass**：
   - 删除 rubric_notes.md 中已被吸收为正式维度的观察
   - 删除已被推翻的假设
   - 更新 rubric_version

## 公众号特有的 Bump 考量

- **粉丝增长因素**：如果两次预测之间粉丝数变化 >20%，阅读量 base 变了，bump 时需标注"base shift"
- **推送时间因素**：公众号推送时间对打开率影响显著（工作日 vs 周末 / 早上 vs 晚上），bump 时需检查样本的推送时间是否可比
- **标题改写因素**：公众号可以修改标题（有限次数），如果用户改过标题，复盘时需把原标题和新标题都记入
