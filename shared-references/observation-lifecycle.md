# 观察生命周期 / Observation Lifecycle

> 原则 #3 的完整规范。rubric_notes.md 不是考古遗址——被推翻的观察要删，被吸收的要升级。

## 核心规则

1. 被新数据推翻的观察 → **删掉**
2. 被吸收为正式维度的观察 → **从"观察记录"段删除，升级到公式段**
3. git history 才是档案——不需要在 markdown 文件里保留"我曾经以为 X，但其实..."
4. 每个观察必须可追溯到具体数据点

## rubric_notes.md 的结构

```markdown
# Rubric Notes — 公众号长文评分规则

## 当前公式

（7 维 + 权重 + composite 计算公式）

## 重大跨样本观察

（≥2 个独立样本支持的 pattern，按置信度排列）

## 观察记录

（单样本观察，按时间倒序。每个观察 ≤3 行）

## 历史公式

（仅记录版本号和变更摘要，不保留完整旧公式——git 里有）
```

## 观察记录的格式

```markdown
### YYYY-MM-DD [文章简称] (id) — [一句话定性]
- 预测：composite=X.XX，bucket=Y，中枢=Z
- 实绩：阅读 N / 分享 S / 在看 K / 留言 M / 完读率 R%
- 判断：哪个维度被验证/推翻？为什么？
- 详见：[predictions/<file>.md]
```

## 升级到"重大跨样本观察"

当一个单样本观察与另一条已有观察形成 ≥2 样本支持时：

```markdown
### Pattern P-01：[一句话命名]
- 样本 1：[文章简称]：[具体发现]
- 样本 2：[文章简称]：[具体发现]
- 可信度：⭐⭐（2/5，基于 2 个独立样本）
- 建议：[formula adjustment]
```

## Cleanup 纪律

每次 bump 结束后，必须执行：
1. 扫描"观察记录"段，删除已被吸收到公式的
2. 扫描"观察记录"段，删除已被新数据推翻的
3. 扫描"重大跨样本观察"，删除 <2 样本支持的（说明样本被推翻）
4. 更新"历史公式"段（仅追加一行版本号 + 摘要）
